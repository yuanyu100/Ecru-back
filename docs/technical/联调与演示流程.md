# 最短联调与演示流程

这份文档的目标不是解释全部模块，而是给出一条最短、最稳定、最适合本地验证和答辩演示的路径。

## 一、启动前检查

先确认以下依赖可用：

1. MySQL 已启动，并存在数据库 `ecru`
2. PostgreSQL 已启动，并存在数据库 `ecru-pg`
3. MinIO 已启动，地址为 `http://localhost:9000`
4. 后端配置文件 `backend/ecru-web/src/main/resources/application.yml` 中的连接信息与你本机一致
5. 如果要演示 AI 对话、AI 图片识别、AI 监控，需提前配置可用的 `AI_API_KEY`、`SILICONFLOW_API_KEY`、`MCP_WEATHER_API_KEY`

如果你只想验证登录、衣橱、后台列表等非 AI 主路径，缺少 AI key 不会阻止系统启动。

建议先执行一次环境自检脚本：

```powershell
.\scripts\check-dev-env.ps1
```

如果当前 PowerShell 执行策略禁止直接运行脚本，改用：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\check-dev-env.ps1
```

它会一次性检查工具、脚本、端口和 AI 环境变量，适合在联调前先排除环境问题。

如果你想把本地开发服务一起拉起，也可以直接执行：

```powershell
.\scripts\start-local-dev.ps1
```

如果当前 PowerShell 提示“禁止运行脚本”，改用：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\start-local-dev.ps1
```

查看状态：

```powershell
.\scripts\status-local-dev.ps1
```

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\status-local-dev.ps1
```

停止：

```powershell
.\scripts\stop-local-dev.ps1
```

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\stop-local-dev.ps1
```

## 二、推荐启动顺序

### 1. 启动依赖服务

如果你的本机已经按仓库脚本约定安装了 MinIO，建议直接用脚本：

```powershell
.\scripts\start-minio.ps1
```

如果当前 PowerShell 执行策略禁止直接运行脚本，改用：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\start-minio.ps1
```

脚本位置：

- [start-minio.ps1](D:/Code/TRAE/bishe/scripts/start-minio.ps1)
- [stop-minio.ps1](D:/Code/TRAE/bishe/scripts/stop-minio.ps1)

### 2. 启动后端

```powershell
cd backend
mvn spring-boot:run -pl ecru-web
```

确认：

- API 地址：`http://localhost:8081/api/v1`

### 3. 启动用户端

```powershell
cd frontend/user-app
npm install
npm run dev
```

### 4. 启动管理端

```powershell
cd frontend/admin-app
npm install
npm run dev
```

说明：

- 两个前端都通过 Vite 代理访问后端
- 常见情况下，先启动的前端占用 `5173`，后启动的会自动切到 `5174` 或其他空闲端口

### 5. 可选：启动演示版 Nginx

如果你想以更接近答辩演示的方式访问前端，可以改用：

```powershell
.\scripts\start-demo-nginx.ps1
```

对应停止脚本：

```powershell
.\scripts\stop-demo-nginx.ps1
```

脚本位置：

- [start-demo-nginx.ps1](D:/Code/TRAE/bishe/scripts/start-demo-nginx.ps1)
- [stop-demo-nginx.ps1](D:/Code/TRAE/bishe/scripts/stop-demo-nginx.ps1)

启动后默认地址：

- 用户端：`http://127.0.0.1:8090/`
- 管理端：`http://127.0.0.1:8090/admin/`
- API 代理：`http://127.0.0.1:8090/api/v1/`

## 四、最短用户端验证路径

### 步骤 1：注册或登录

进入用户端：

- 注册新账号，或直接使用 `testuser / 123456`

验证点：

- 能成功进入首页
- 个人页能看到当前用户信息

### 步骤 2：录入衣物

进入“衣橱”或“新增衣物”页面，完成至少 1 件衣物录入。

建议优先走这两种方式之一：

1. 手动填写衣物信息并上传图片
2. 如果 AI key 正常，演示 AI 识别后回填衣物信息

验证点：

- 新衣物出现在衣橱列表
- 可进入衣物详情页
- 可编辑后再次保存

如果想用 HTML 手工页辅助联调，可打开：

- `backend/manual-tests/upload-clothing.html`

## 五、最短 AI 演示路径

### 步骤 3：发起 AI 穿搭对话

进入用户端“穿搭助手”页面，输入一段明确场景描述，例如：

```text
明天上海 20 度，通勤场景，帮我搭一套不显臃肿的穿搭
```

如果 AI 能力和天气能力都正常，系统会结合：

- 当前用户衣橱
- 对话内容
- 天气信息
- 风格偏好

生成回复和穿搭建议。

验证点：

- 对话页能收到 AI 回复
- 首页“最近会话”有数据
- 首页“最近穿搭记录”有新记录

### 步骤 4：进入穿搭历史详情页

从首页最近穿搭记录进入详情页，验证这几个动作：

1. 查看推荐单品和理由
2. 收藏或取消收藏
3. 提交评分与文字反馈
4. 删除当前记录

验证点：

- 收藏状态切换成功
- 反馈保存后刷新仍在
- 删除后返回首页，记录消失

## 六、最短管理端验证路径

### 步骤 5：登录管理端

使用管理员账号登录管理端。

验证点：

- 能进入 Dashboard
- 左侧能看到“用户管理”和“AI 监控”

### 步骤 6：检查用户和衣物后台

先看“用户管理”：

- 能看到用户列表
- 能执行启用 / 禁用

再看“衣物台账”：

- 能看到衣物列表
- 管理员视角可以按所属用户查看
- 能删除衣物记录

## 七、AI 监控演示路径

### 步骤 7：先在用户端触发 AI 调用

建议先完成一次：

1. AI 对话
2. AI 穿搭建议
3. AI 图片识别

### 步骤 8：回到管理端查看 AI 监控

进入“AI 监控”页面，确认以下内容有数据：

- 今日调用总数
- 成功率
- 平均响应时间
- 今日 Token 消耗
- 7 天趋势
- 今日 24 小时趋势
- 按场景分布
- 按模型分布
- 最近调用记录
- 最近 24 小时错误分布

如果刚触发过 AI 请求，监控页刷新后应该能立即看到最新调用；小时级和日级统计会由定时任务继续沉淀。

## 八、答辩时推荐演示顺序

如果你只有几分钟，推荐按这个顺序：

1. 展示用户端登录成功
2. 展示衣橱已有衣物
3. 发起一次 AI 穿搭请求
4. 回到首页展示最近会话和最近穿搭记录
5. 打开穿搭历史详情，演示收藏和反馈
6. 切到管理端，展示用户列表和衣物台账
7. 最后展示 AI 监控页的调用记录和趋势

这条链路覆盖了：

- 用户
- 衣物
- AI
- 历史记录
- 后台管理
- 监控

## 九、出问题时先查哪里

### 用户端登录失败

优先检查：

- 后端是否启动
- MySQL 是否可连接
- `/api/v1/auth/login` 是否返回 200

可用素材：

- `backend/manual-tests/login.html`
- `backend/manual-tests/test-login.js`

### 衣物上传失败

优先检查：

- MinIO 是否启动
- `minio.bucketName` 是否存在
- 图片 URL 是否能回显

可用素材：

- `backend/manual-tests/upload-clothing.html`
- `backend/manual-tests/test-minio-access.ps1`
- `scripts/start-minio.ps1`

### AI 没有返回

优先检查：

- `AI_API_KEY`
- `SILICONFLOW_API_KEY`
- `MCP_WEATHER_API_KEY`
- 外部网络是否可访问对应服务

### 管理端看不到管理菜单

优先检查：

- 当前登录用户是否为 `ADMIN`
- `upgrade-user-role-and-admin-clothing.sql` 是否已经执行

## 十、建议你本地固定保留的演示数据

为了避免每次重置数据库后都要重新准备，建议至少保留：

- 1 个普通用户
- 1 个管理员
- 每个用户至少 2 到 3 件衣物
- 至少 1 条 AI 会话记录
- 至少 1 条穿搭历史记录
- 至少 1 次失败的 AI 调用记录

这样无论演示用户端还是管理端，页面都不会是空的。
