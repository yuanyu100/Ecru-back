# TRAE 项目全流程执行规则（Java SpringBoot 多模块精简版）
## 文档说明
适配Java SpringBoot多模块架构，核心定**最小可行约束规则**，聚焦**兼容性、可追溯、防线上故障**，所有开发/测试/运维人员严格遵守，轻量落地、高效执行。

## 一、核心开发三原则
1. **增量扩展**：只新增、不直接修改/删除原有代码、配置、库字段、接口，跨模块变更不侵入无关联模块。
2. **兼容兜底**：变更需保留旧逻辑完整兼容，明确**兼容下线时间（≥7天，跨模块≥14天）**，全链路验证无依赖后再移除。
3. **可回滚**：所有变更留痕，发布前准备**模块级/全项目回滚预案**，确保10分钟内可恢复。

## 二、代码规范（核心约束）
### 2.1 基础要求
1. 遵循**Alibaba Java开发手册**，通过CheckStyle/Spotless做格式校验，不通过禁止合入。
2. 核心接口/方法/库实体/迁移脚本加**关键注释**：功能、边界、变更目的，跨模块Feign接口/Javadoc注释完整。
3. 入参必校验：后端用Jakarta Validation（@NotBlank/@NotNull），封装全局统一异常返回格式，无原生异常抛至前端。
4. 外部脚本统一放`/scripts/`，按指定格式输出结果、过程日志化，禁止环境变量/全局变量，配置内聚在脚本中。

### 2.2 兼容性核心规则（强制执行）
1. **方法/接口**：不修改原有入参/返回值/名称，优化则新增带版本后缀的方法/接口（如`getUser→getUserV2`），旧方法标注**变更标识+兼容下线时间**；HTTP接口通过URL版本隔离（`/api/v1/→/api/v2/`）。
2. **类/字段**：不修改字段类型/名称，调整则新增字段并做**双读双写**（写同步新旧、读优先新字段），旧字段加@Deprecated并标注**废弃下线时间+新字段映射关系**。
3. **枚举/配置**：不修改/删除原有枚举值/配置项，新增追加至末尾；配置只追加、不覆盖，核心配置变更需审批。

## 三、多模块项目结构规范
1. 按**common→api→business→gateway**分层，模块间**无循环依赖**，跨模块调用仅通过api层Feign接口实现。
2. 每个业务模块内建`compatibility`包，存放本模块兼容逻辑；跨模块兼容逻辑统一放**全局compatibility模块**，避免兼容代码散落。
3. 包/文件夹层级**≤4层**，根目录维护**极简CHANGELOG.md**：记录版本、变更人、核心变更、兼容下线时间。
4. 根pom统一管理依赖版本，子模块仅引入不指定版本，禁止跨模块直接依赖业务实现层。

## 四、配置与依赖管理
### 4.1 配置
1. 按**dev/test/prod**环境隔离，敏感配置（密码/密钥）通过配置中心/Nacos注入，**禁止硬编码**。
2. 配置变更**只追加不修改/删除**，核心配置（库连接/功能开关/限流阈值）变更需提交审批，配置文件头部标注**版本+变更标识**。
3. 配置读取加**默认值兜底**（如@Value("${key:default}")），启动时校验核心配置，缺失告警并加载兜底值。

### 4.2 依赖
1. 禁止直接升级依赖**主版本**（如2.x→3.x），升级需单独测试、全链路验证无兼容问题后合入。
2. 新增第三方依赖需**简单审批**（说明用途/维护状态），优先选择成熟、无安全漏洞的依赖；定期清理无用依赖。
3. 开发环境使用隔离环境，避免全局依赖污染，禁止system范围依赖。

## 五、测试与CI/CD校验（极简要求）
1. **基础测试**：核心业务逻辑单元测试覆盖率**≥70%**，跨模块核心接口**≥90%**，用JUnit5+Mockito做Mock，覆盖正常/异常/边界场景。
2. **兼容性测试**：新增功能后**全量回归原有测试用例**，必须100%通过；库变更验证原有SQL/查询无异常，接口验证旧版本契约不被破坏。
3. **CI/CD自动阻断**：代码提交后自动执行**代码规范+单元测试+静态扫描（Sonar）**，任意步骤失败，直接阻断合入，无人工干预余地。

## 六、数据库操作规范（红线规则）
1. 禁止直接删除/修改库表/字段/约束，废弃字段先标注**废弃注释+下线时间**，确认无数据依赖后再删除。
2. 字段类型/长度调整，按**新增字段→双读双写→切换新字段→删除旧字段**执行，每步测试环境验证。
3. 迁移脚本（Flyway/Liquibase）**必含回滚逻辑**，执行前做dry-run，由指定运维执行，执行前全量备份库数据。

### 6.1 PostgreSQL 和 MySQL 混用注意点
1. **驱动依赖管理**：同时添加 PostgreSQL 和 MySQL 驱动依赖，且都设置为 runtime 作用域，避免编译期冲突。
2. **数据源配置隔离**：使用不同的配置前缀（如 `spring.datasource` 和 `spring.datasource-pg`）隔离两个数据源的配置。
3. **连接池配置**：为每个数据源单独配置 Hikari 连接池参数，避免连接池参数冲突。
4. **SQL 语法兼容**：编写 SQL 时注意两个数据库的语法差异，优先使用标准 SQL，避免使用数据库特有语法。
5. **事务管理**：多数据源场景下使用 JTA 或手动管理事务，确保跨数据库操作的一致性。
6. **性能监控**：为两个数据库分别配置独立的监控指标，及时发现性能瓶颈。
7. **备份策略**：为两个数据库制定独立的备份策略，确保数据安全。

## 七、发布与回滚规则
1. **灰度发布**：所有新功能通过**配置中心功能开关**控制，默认关闭；先灰度10%实例/用户，监控2小时核心指标（报错率/响应时间/慢查询），无异常再全量开启。
2. **回滚准备**：发布前备齐**上一版本代码包+配置备份+库全量备份**，编写《极简回滚手册》，确保专人10分钟内完成回滚。
3. **线上监控**：新功能上线后24小时重点监控**旧接口/旧业务**指标，异常立即关闭功能开关并执行回滚。

## 八、违规处理（极简落地）
1. 违反兼容性/库操作规则，CI/CD直接阻断合入，整改后重新提交；
2. 未做回滚预案/未通过兼容性测试，驳回发布申请；
3. 因违规导致线上事故，组织复盘并优化规则，责任人承担对应责任。
